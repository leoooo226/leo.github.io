    Lane¶

    lane.sv¶

    lane的顶层模块，例化了lane的各个子模块，如lane_sequencer、operand_requester、VRF、Operand queues、Vector Functional Units等。

    lane_sequencer¶

    模块声明¶

    lane_sequencer 模块通过导入不同的包（如 ara_pkg 和 rvv_pkg）来获得需要的类型和定义。定义了一些参数（如 NrLanes）和输入输出端口。

    请求注册¶

    使用了寄存器处理指令的请求，确保不会重复采样相同指令。通过 last_id存储上一个指令的ID。

    操作请求命令队列¶

    维护指令执行所需的操作数请求，由于指令之间可能存在依赖关系，因此需要谨慎处理请求。

    VFU操作控制¶

    • 向量指令的执行状态跟踪，通过vinsn_done_d和vinsn_running_d数组来跟踪哪些指令已完成或正在执行。
    • 向量功能单元（VFU）操作的准备,vfu_operation_d结构体包含了即将发送到VFU的操作的详细信息，包括指令ID、操作类型、向量掩码等。
    • 6个VFU: VFU_Alu、VFU_MFpu、VFU_LoadUnit、VFU_SlideUnit、VFU_StoreUnit、VFU_MaskUnit
    • 9个操作数队列：ara_pkg: AluA, AluB, MulFPUA, MulFPUB, MulFPUC, MaskB, MaskM, StA, SlideAddrGenA
    • 条件执行：根据VFU的准备情况、操作请求者的状态以及指令的特性（如vl为零，mask的vl特殊vl: (pe_req.vl / NrLanes / 8) >> unsigned(pe_req.vtype.vsew)）来决定是否执行某些操作。
        if (pe_req_valid) begin
      unique case (pe_req.vfu)
        VFU_Alu : begin
          pe_req_ready = !(operand_request_valid_o[AluA] ||
            operand_request_valid_o[AluB ] ||
            operand_request_valid_o[MaskM]);
        end
        VFU_MFpu : begin
          pe_req_ready = !(operand_request_valid_o[MulFPUA] ||
            operand_request_valid_o[MulFPUB] ||
            operand_request_valid_o[MulFPUC] ||
            operand_request_valid_o[MaskM]);
        end
        VFU_LoadUnit : pe_req_ready = !(operand_request_valid_o[MaskM] ||
            (pe_req_i.op == VLXE && operand_request_valid_o[SlideAddrGenA]));

        VFU_SlideUnit: pe_req_ready = !(operand_request_valid_o[SlideAddrGenA]);

        VFU_StoreUnit: begin
          pe_req_ready = !(operand_request_valid_o[StA] ||
            operand_request_valid_o[MaskM] ||
            (pe_req_i.op == VSXE && operand_request_valid_o[SlideAddrGenA]));
        end
        VFU_MaskUnit : begin
          pe_req_ready = !(operand_request_valid_o[AluA] ||
            operand_request_valid_o [AluB] ||
            operand_request_valid_o [MulFPUA] ||
            operand_request_valid_o [MulFPUB] ||
            operand_request_valid_o[MaskB] ||
            operand_request_valid_o[MaskM]);
        end
        VFU_None : begin
          pe_req_ready = !(operand_request_valid_o[MaskB]);
        end
        default:;
      endcase
    end
    • vstart/vl
      for element index x
  prestart(x) = (0 <= x < vstart)
  body(x) = (vstart <= x < vl)
  tail(x) = (vl <= x < max(VLMAX,VLEN/SEW))
  mask(x) = unmasked || v0.mask[x] == 1
  active(x) = body(x) && mask(x)
  inactive(x) = body(x) && !mask(x)

    处理指令¶

    always_comb 和 always_ff 块用于实现组合逻辑和时序逻辑，处理请求的有效性、指令的调度和执行。 根据指令的类型（VFU类型），做出不同的操作数请求。

    VFU操作和请求的具体实现¶

    根据操作的不同（如 ALU、MFPU、Load、Store），设置相应的操作数请求，确保正确传递信息。

    operand_requester¶

    VRF¶

    Ara的向量寄存器文件（VRF）使用一组单端口（1RW）存储bank实现。每个bank的宽度是64位，其与每个通道的数据路径的宽度相同。每个通道有八个bank，导致每个通道有八个单端口存储bank。在RVV中有32个向量寄存器。每个向量寄存器的大小（以位为单位）取决于实现（最小值= 128位）

    跨通道拆分VRF的优势：

    • 由于通道是处理元件（PE），因此大多数计算都限制在一个通道内，这使得硬件实现变得容易。
    • 消除了执行单元对通道数量的依赖性，这使得Ara具有可扩展性。
    • 由于VRF在通道内部，因此VRF和通道之间的路由执行逻辑被简化

    通道内的VRF部分被进一步划分为八个64位宽的bank。每个bank使用单端口SRAM存储器。作为一个单端口存储器，它只有一个地址端口用于读取和写入。在稳定状态下，在最坏情况条件下，同时访问5个存储体以支持乘法累加指令（需要3个源寄存器加上掩码寄存器，导致4次读取，以及一个目的地寄存器，导致单次写入）。这是单个指令的最坏情况。但是可以在不同的单元中同时执行更多的指令（例如vload + vslide + vmacc.m），并且它们可以同时访问每个通道最多8个存储体。

    于每个通道具有4KiB的VRF存储器，因此去往每个通道的地址总线将为12位宽。由于通道内有8个库，所以地址的最低有效3位用于寻址8个库，并且剩余9位用于寻址库内的字节。在这些剩余的9位中，6位用于寻址存储体内的64个64位位置中的1个，并且地址的剩余3位用于寻址64位内的单个字节。

    Translating Vector Register Number to SRAM memory Bank¶

    向量寄存器编号字段的值范围为0到31（5位），存在于指令中。它们被解码并用于确定VRF存储器中该向量寄存器的起始地址。在通道内，向量寄存器被存储为连续的字节集。寄存器号（vid）乘以一个因子，该因子等于单个向量寄存器中的字节数除以通道数。得到的数字给出了VRF存储器中向量寄存器的开始地址。

    Organization of Elements in Vector Register File¶

    向量寄存器的元素被映射到VRF中的连续通道。每个通道的数据路径宽度为64位，与单个寄存器堆组的宽度相匹配。下图显示了4个通道，编号为0-3，每个通道有8个字节（64位），连续的字节编号为0 - 1F。由于每个通道中存在功能单元，因此可以并行处理元素组。例如，如果我们有四个64位元素和四个通道，那么每个元素都被放置在下面数字0-3所示的不同通道中。所有这4个元件都可以从VRF同时访问，并且可以并行操作。

    继续先前的跨通道的布置，单个通道内的元素也以一定顺序布置，使得当元素宽度改变时，一些元素和通道之间的映射保持相同。因此，通道中的连续字节与元素的连续字节之间的映射不相同。这可以通过使用下一节中描述的混洗逻辑来解决。

    作为具有256 B（2048位）的向量长度和4个通道（每个通道是64位宽）的示例，用于各种大小的SEW的字节的映射如下所示：

    对于第一种情况（SEW = 64），这种安排简单而明显。每个64位元素被映射到不同的通道中，并且每个元素的字节被连续地存储在通道内。对于第二种情况（SEW = 32），元素0、1、2和3被映射到通道0到通道3的下半部分（32位）。然后，元素4被映射到通道0的上半部分，依此类推，直到元素7被映射到通道3的上半部分。重要的是要注意，对于SEW的两种情况，元素0、1、2和3的最低有效字节被映射到对应通道内的相同字节位置。对于SEW的剩余值继续重复该过程。

    在SEW = 16和8的情况下，还有一件事需要观察。当我们从通道0开始分配新元素的第一个字节时，我们还观察通道中的最后一个元素被分配了通道的哪一半（高或低32位）。对于新的分配，我们选择上一次没有使用的那一半。因此，在SEW = 16的情况下，当分配元素8时，它被放置在通道0的字节位置2（通道0的下半部分）而不是字节位置6（通道0的上半部分），因为通道0中分配的最后一个元素，元素4，被分配在字节位置4（通道0的上半部分）。

    VRF中的元素及其对应字节的上述布置被约束在通道内。通道外的数据（即内存中的数据）按照递增的内存地址从最低有效字节到最高有效字节简单打包。因此，当数据在存储器和VRF之间移动时，字节被混洗/解混洗以匹配适当的所需排序。元素和字节的这两种排序被称为字节的通道组织和字节的自然打包。

    通道组织：元素及其对应字节存储在VRF中的方式，如上所述。

    自然打包：在内存中打包的元素和字节，最低有效字节与最高有效字节在内存地址中递增。

    洗牌逻辑¶

    如下图所示，混洗/解混洗逻辑位于内存子系统和VRF之间。当数据从存储器移动到VRF（通过加载指令）时，它会从自然打包排列混洗到通道组织排列。类似地，当数据从VRF移动到内存（通过存储指令）时，它会从通道组织（Lane Organization）到自然打包（Natural Packing）排列进行解混洗。

    对于4个通道和16个SEW，从自然打包到通道组织的字节映射如下图所示。对于元素0，字节索引是相同的，映射到VRF中的字节索引0和1，用于自然包装和通道组织。元素1被映射到VRF中的字节索引8，其两个字节在VRF中的索引8和9中。混洗逻辑从存储器中取出顺序字节，如自然打包行中所示，并将其转换为通道组织行中所示的通道组织排列。反混洗逻辑则相反。

    混洗/解混洗逻辑的布置是向量的SEW的函数。这意味着当一个向量从内存移动到VRF时，字节会根据向量的SEW值进行混洗。类似地，当向量从VRF移动到存储器时，字节基于向量的SEW被重新洗牌。因此，除了存储在VRF中的向量的字节之外，它还在硬件中用其SEW进行标记。当数据被移回内存时，该标签随后被解混洗逻辑使用。

    下图显示了字节如何存储在内存中（自然打包）以及它们如何存储在VRF（通道组织）中。名为A0到H3的32个字节按顺序存储在内存位置0到31（自然顺序），如下面的内存图像所示。随后，这32个字节使用不同的SEW值存储在VRF中，如下面的VRF图像所示。

    例如，通道1的字节位置11可以具有分别用于SEW值64、32、16和8的存储器字节C3、B3、E3或G1。所以这个字节的shuffle多路复用器如下：

    Lane 1 [ location 11] = (SEW = = 64) ? C3 : ((SEW == 32) ? B3 : ((SEW == 16) ? E3 : G1))

    类似地，用于存储单元11的deshuffle多路复用器如下：

    Memory [location 11] = (SEW = = 64) ? Lane 1[location 11] : ((SEW == 32) ? Lane 2[location 19] : ((SEW == 16) ? Lane 1[location 13] : Lane 3[location 25]))

    这些字节多路复用器在两个方向上被复制，每个字节从存储器进入VRF，每个字节从VRF出去到存储器。

    Reshuffle Logic¶

    在具有跨多个通道分布的VRF的多通道架构中，对于跨通道边界的指令出现某些特殊问题（例如，缩小和扩大指令）。由于支持尾部不受干扰策略的架构要求，这些问题进一步加剧，即，不应修改超过向量长度的元素的要求。当指令写入向量寄存器vd时，该向量寄存器最初用与新EEW不同的EEW编码，并且vd的旧内容未被完全重写，则vd的旧内容被破坏，并且尾部不受干扰策略被违反。为了避免尾部元素的这种损坏，可以访问所有通道的滑动单元对向量寄存器执行混洗操作。这种洗牌操作称为重新洗牌。

    让我们以具有256位（32 B）宽的向量寄存器和四个通道的简单向量单元为例，其中每个通道为64位（8 B）宽。每个通道都有一个256 B SRAM存储体，配置为32x64位宽的1R1W端口存储器。因此，每个bank（这里每个通道只有一个bank）都实施了VRF的四分之一。以下示例限于SEW值16和32。
